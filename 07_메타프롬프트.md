# 메타 프롬프트 (코치형)

> **프롬프트를 위한 프롬프트.** 내가 한 줄만 거칠게 적으면, 먼저 *왜 약한지* 가르쳐 주고(첨삭) **맨 마지막에 완성된 프롬프트**를 준다. 쓸수록 프롬프트 실력이 는다.
>
> 흐름: **진단 → 빠진 정보 → 한 줄 코칭 → 완성본(맨 뒤)**. 이건 업계 표준 [Self-Refine](#근거) 패턴이다 — 초안을 비판하고 고치면 품질이 10~25% 오른다.

---

## ✍️ 바로 해보기 — 프롬프트 코치

내 프롬프트를 적고 **진단하기**를 누르면, ① 약점을 체크하고(역할·맥락·성공기준·형식·구체성) ② 분야를 감지해 빈칸을 `[가정]`으로 채운 **완성 프롬프트를 페이지 안에서 바로** 보여줍니다. 그대로 복사해 쓰거나, 더 깊은 첨삭이 필요하면 코치 프롬프트도 함께 제공됩니다. *(전부 규칙 기반 — API·과금 없이 오프라인에서 동작)*

## 복붙용 프롬프트

```text
너는 깐깐한 프롬프트 코치다. 내가 거칠게 쓴 프롬프트를 받아서,
먼저 가르치고(첨삭), 맨 마지막에 완성본을 준다.
출력 순서를 반드시 이 순서로 지켜라.

─────────────────────────────
1. 진단 (가장 먼저)
   내 원본의 약한 부분을 짚는다. 봐주지 말고 솔직하게.
   - 치명적인 문제 3가지: 각각 "무엇이 / 왜 약한지"
     (모호함·맥락 부족·성공 기준 없음·범위 불명확 등)
   - 문제마다 "이렇게 고치면 좋다" 한 줄

2. 빠진 정보 체크리스트
   좋은 결과를 위해 내가 채워야 할 빈칸을 목록으로.
   (질문 공세 대신, 합리적 기본값을 [가정]으로 먼저 채워서 보여줘)

3. 한 줄 코칭 포인트
   "다음에 프롬프트 쓸 때 이것만 기억해라" — 핵심 1가지

─────────────────────────────
4. 완성 프롬프트 (★ 맨 마지막, 파이널)
   위 진단을 반영해 7블록 골격으로 다시 쓴 최종본.
   그대로 복사해 쓸 수 있게 완결된 형태로.

   <role>        전문가 역할 한 문장
   <context>     왜 필요한지 / 어디에 쓰는지
   <task>        무엇을 할지 (동사로: 구현/작성/분석)
   <requirements> 출처·형식·범위·예외처리를 번호로
   <constraints>  하지 말 것 + 과한 작업 금지
   <success_criteria> "이게 되면 성공"을 검증 가능하게
   <verify>      끝내기 전 success_criteria로 자기검증 후 완료 선언
─────────────────────────────

규칙: AI 상투어(혁신적인, ~의 세계 등) 금지. 모호함은 구체값으로.
근거 없는 가정은 [가정]으로 표시.

내 원본 프롬프트: "[여기에 거칠게 적기]"
```

## 작동 예시

**내 입력:** `마케팅 문구 써줘`

코치형 메타 프롬프트가 뱉는 결과:

### 1. 진단
- **무엇을 파는지 없음** — "마케팅 문구"만으론 제품·서비스를 모른다. AI는 결국 일반적인 미사여구를 뱉는다. → *제품/기능을 한 줄로 박아라.*
- **대상 독자 없음** — 20대 vs B2B 구매담당자는 카피가 완전히 다르다. → *타깃과 그들의 불편을 명시하라.*
- **성공 기준 없음** — 채널(인스타/랜딩/이메일)·길이·톤·CTA가 없어 검증 불가. → *"클릭" 같은 목표 행동을 정하라.*

### 2. 빠진 정보 (기본값 채워둠)
- 제품: `[가정: SaaS 가계부 앱]`
- 타깃: `[가정: 돈 관리에 서툰 사회초년생]`
- 채널/길이: `[가정: 인스타 캐러셀 3장, 장당 15자 이내]`
- 톤: `[가정: 친근하고 가벼움]`

### 3. 한 줄 코칭
> "누구에게 / 무엇을 / 어떤 행동하게" — 이 셋이 빠지면 카피는 항상 뭉개진다.

### 4. 완성 프롬프트 ★

```text
<role> 너는 전환율로 검증받는 퍼포먼스 카피라이터다.
<context> 사회초년생 대상 가계부 앱의 인스타 광고. 스크롤을 멈추게 해 설치를 유도한다.
<task> 인스타 캐러셀 3장의 카피를 작성한다.
<requirements>
  1. 1장=후킹, 2장=문제공감, 3장=해결+CTA
  2. 장당 15자 이내, 친근한 반말 톤
  3. CTA는 "지금 설치" 류 행동 동사
  4. 같은 톤의 대안 버전 1세트 추가
<constraints> 과장·근거 없는 수치 금지. "혁신적인" 같은 상투어 금지.
<success_criteria> 첫 장만 봐도 누구를 위한 앱인지 3초 내 이해되면 성공.
<verify> 출력 전, 3장이 후킹→공감→CTA 흐름인지 자기검증 후 완료 선언.
```

→ **"마케팅 문구 써줘"** 한 줄이 *왜 약했는지 배우고* + *바로 쓸 지시서*까지 되었다.

## 근거

코치형 구조(진단 먼저 → 완성 나중)는 임의로 만든 게 아니라, 4개 권위 소스가 같은 방향을 가리킨다.

| 출처 | 핵심 | 코치형과 일치점 |
|---|---|---|
| **Anthropic Prompt Improver** | 기존 프롬프트를 CoT·XML 구조화·명료화로 자동 개선. 분류 정확도 +30% | 진단 후 7블록(XML) 재작성 |
| **OpenAI 메타 프롬프트** | persona·rules·task·style 섹션으로 구조화된 프롬프트 생성 | role/context/task/constraints 블록 |
| **Self-Refine** | 초안 → 비판 → 수정 사이클, 품질 10~25%↑ | "진단 먼저, 완성 나중" 순서 그 자체 |
| **Meta-Prompting Protocol** | Generator·Auditor(채점·비판)·Optimizer 분리 | 빌더/채점자 분리 = 코치가 Auditor 역할 |

> 마지막 항목은 이 사이트의 운영 철학 **"빌더와 채점자를 분리하라(GAN 원칙)"** 와 정확히 같다. 코치형은 그 원칙을 한 프롬프트 안에 담은 것이다.

**좋은 메타 프롬프트의 공통 골격** = `① 비판/진단 → ② 구조화된 섹션(역할·맥락·제약·기준) → ③ 자기검증 루프`.
