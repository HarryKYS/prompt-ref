## ⭐ 0. 메타 프롬프트 (코치형) — 막힐 때 제일 먼저

*뭘 써야 할지 모를 때, 한 줄만 적으면 약점을 짚어주고 완성된 프롬프트까지 만들어 준다. 자세한 예시·근거는 [07 메타 프롬프트](/07_메타프롬프트).*

<div data-pc-no-prac></div>

```text
너는 깐깐한 프롬프트 코치다. 내가 거칠게 쓴 프롬프트를 받아서,
먼저 가르치고(첨삭), 맨 마지막에 완성본을 준다. 이 순서를 지켜라.

1. 진단 — 원본의 치명적 약점 3가지: "무엇이 / 왜 약한지" + 고치는 법 한 줄
2. 빠진 정보 — 채워야 할 빈칸 목록 (기본값은 [가정]으로 먼저 채워서)
3. 한 줄 코칭 — "다음엔 이것만 기억해라" 핵심 1가지
4. 완성 프롬프트 (★맨 마지막) — 7블록으로 다시 쓴 최종본:
   <role> <context> <task> <requirements> <constraints> <success_criteria> <verify>

규칙: AI 상투어 금지. 모호함은 구체값으로. 가정은 [가정] 표시.
내 원본 프롬프트: "[여기에 거칠게 적기]"
```

## 1. 문서 요약 (핵심만)

```text
너는 핵심을 놓치지 않는 에디터다. 아래 글을 다음 형식으로 요약해줘.

1. 한 줄 요약: 전체를 한 문장으로
2. 핵심 포인트: 가장 중요한 3~5개를 불릿으로
3. 시사점/액션: 그래서 무엇을 하면 되는지

원문에 없는 내용은 절대 지어내지 말고, 불확실한 부분은 "원문에 명시 안 됨"이라고 표시해.

[여기에 원문 붙여넣기]
```

## 2. 글 다듬기 (톤 유지)

```text
아래 글을 더 명확하고 읽기 쉽게 다듬어줘. 단, 내 원래 어조와 의도는 그대로 유지하고 의미를 바꾸지 마.

- 중복·군더더기 제거, 한 문장은 한 가지만
- 수동태→능동태, 모호한 표현은 구체적으로
- 무엇을 왜 고쳤는지 끝에 3줄로 요약

[여기에 글 붙여넣기]
```

## 3. 7블록 작업 골격 만들기 ★

```text
내가 하려는 작업을 아래 7블록 골격으로 구조화해줘. 빈칸은 합리적으로 추론해 채우고, 정말 모르는 것만 내게 질문해.

<role> 이 작업을 수행하는 전문가 역할 한 문장
<context> 왜 필요한지 / 어디에 쓰는지
<task> 무엇을 할지 (제안이 아니라 구현/실행으로)
<requirements> 데이터 출처·기술스택·형식·에러처리를 번호로
<constraints> 오버엔지니어링 금지 + 내 제약
<success_criteria> "이게 되면 성공이다"를 검증 가능하게
<verify> 끝내기 전 success_criteria로 자기검증 후 완료 선언

내 작업: [여기에 한 줄로]
```

## 4. PRD 초안 (제품 요구사항)

```text
너는 실리콘밸리 PM이다. 아래 아이디어를 PRD로 정리해줘.

1. 문제 정의 — 누가, 어떤 상황에서, 무엇이 불편한가
2. 목표 & 성공 지표 — 측정 가능하게
3. 핵심 사용자 시나리오 — 3개
4. 기능 요구사항 — 우선순위(P0/P1/P2)와 함께
5. 비목표(이번엔 안 하는 것)
6. 리스크 & 미해결 질문

근거 없는 가정은 "[가정]"으로 표시하고, 내게 되물어야 할 질문은 끝에 모아줘.

아이디어: [여기에]
```

## 6. 코드 리뷰 (시니어 관점)

```text
너는 까다로운 시니어 엔지니어다. 아래 코드를 리뷰해줘.

심각도(CRITICAL/HIGH/MEDIUM/LOW)로 분류하고, 각 항목에:
- 무엇이 문제인지
- 왜 문제인지 (버그/보안/성능/가독성)

보안(입력 검증·인젝션·시크릿)과 버그를 먼저, 스타일은 마지막에. 문제 없으면 "문제 없음"이라고 솔직히 말해.

[코드 붙여넣기]
```
