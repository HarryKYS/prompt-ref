# 프롬프트 체크리스트

## 작성 전 (3가지 전제)

- [ ] **성공 기준**을 한 문장으로 적었다 ("이게 되면 성공이다")
- [ ] 그 기준을 **테스트할 방법**이 있다 (눈으로 확인할 산출물, 비교 대상)
- [ ] 개선할 **초안**이 있다 (백지에서 시작 안 함)

## 작성 중

- [ ] 원하는 **출력 형식·제약**을 구체적으로 명시했나
- [ ] 순서가 중요한 부분을 **번호 목록**으로 나눴나
- [ ] 지시의 **이유**(맥락)를 한 줄이라도 줬나
- [ ] 형식·톤이 중요하면 **예시 3~5개**를 `<example>`로 넣었나
- [ ] 지시·맥락·입력이 섞였다면 **XML 태그**로 구분했나
- [ ] 긴 데이터(20k+)는 **맨 위**에 두고 질문은 아래에 뒀나
- [ ] 필요하면 **역할**을 시스템 프롬프트에 한 줄 줬나
- [ ] "하지 마"가 아니라 **"이렇게 해"로** 썼나

## 작성 후 (자기검증)

- [ ] 맥락 없는 사람이 읽어도 안 헷갈리나 (**황금률**)
- [ ] 템플릿을 그대로 복붙한 빈칸 <code v-pre>{{...}}</code>이 남아있지 않나
- [ ] 결과를 **검증·자기교정**하라는 지시를 넣었나
- [ ] (최신 모델) 과거에 넣던 "CRITICAL/MUST/반드시" 같은 **과잉 압박어**가 남아 과잉행동을 유발하진 않나 → [03_최신모델_주의점.md](03_최신모델_주의점.md)

## 결과가 어긋날 때 디버깅 순서

1. 출력 형식이 안 맞음 → 예시 추가 + "할 것"으로 재지시 + 프롬프트 스타일을 출력 스타일과 일치
2. 행동을 안 하고 제안만 함 → "수정해/구현해"로 명시 (도구 사용 명령)
3. 너무 장황/과잉설계 → "최소한으로, 요청한 것만" 제약 추가
4. 코드 추측·환각 → "파일 먼저 읽고 답하라" 지시
5. 추론이 부족 → "충분히 생각하고, 끝내기 전 검증하라"
