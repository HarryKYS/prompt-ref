# 핵심 원칙 — 전 모델 공통

> Claude를 "똑똑하지만 우리 회사 맥락을 전혀 모르는 신입"이라고 생각하라.
> **황금률:** 맥락 없는 동료에게 프롬프트를 보여주고 따라하게 했을 때 헷갈린다면, Claude도 헷갈린다.

---

## 명확하고 직접적으로 (Be clear and direct)

- 원하는 출력 형식·제약을 구체적으로 명시한다.
- 순서·완전성이 중요하면 **번호 매긴 목록**으로 지시한다.
- "기대 이상"을 원하면 그렇게 **명시적으로 요청**한다. 알아서 해주길 기대하지 말 것.

```
❌  분석 대시보드 만들어줘
✅  분석 대시보드 만들어줘. 관련 기능과 인터랙션을 최대한 많이 포함하고,
    기본을 넘어 완전히 작동하는 구현으로 만들어줘.
```

## 맥락(이유)을 줘라 (Add context)

지시의 **이유**를 설명하면 Claude가 일반화해서 더 정확히 따른다.

```
❌  말줄임표(…) 절대 쓰지 마
✅  이 응답은 TTS 엔진으로 읽힐 거야. 말줄임표는 발음 처리를 못 하니 쓰지 마.
```

## 예시를 줘라 (Multishot / Few-shot)

출력 형식·톤·구조를 잡는 **가장 신뢰도 높은 방법**. 3~5개가 적정.

- **관련성:** 실제 쓸 케이스를 그대로 닮게
- **다양성:** 엣지 케이스 포함, 의도치 않은 패턴을 학습하지 않게 변주
- **구조화:** `<example>` 태그로 감싸 지시문과 구분 (여러 개는 `<examples>`)

> 팁: Claude에게 "이 예시들이 관련성·다양성 충분한지 평가해줘"라고 시킬 수도 있다.

## XML 태그로 구조화 (Structure with XML)

지시·맥락·예시·입력이 섞일 때 태그로 감싸면 오해가 준다.

- 일관되고 설명적인 태그명 사용 (`<instructions>`, `<context>`, `<input>`)
- 계층이 있으면 중첩 (`<documents>` 안에 `<document index="n">`)

## 역할을 줘라 (Role / System prompt)

시스템 프롬프트에 역할 한 문장만 넣어도 톤과 행동이 달라진다.

```python
system="You are a helpful coding assistant specializing in Python."
```

## 긴 컨텍스트 다루기 (Long context, 20k+ 토큰)

- **긴 데이터는 맨 위에**, 질문·지시·예시는 그 아래에. (응답 품질 최대 +30%)
- 여러 문서는 `<document>` 안에 `<document_content>` + `<source>` 메타로 감싼다.
- **인용 먼저:** "관련 부분을 `<quotes>`로 먼저 뽑은 뒤 그걸 근거로 답하라" → 노이즈 제거.

## 출력 형식 제어 (Control format)

1. **하지 말 것보다 할 것을 지시:** "마크다운 쓰지 마" ❌ → "흐르는 산문 문단으로 써" ✅
2. **XML 형식 지시자:** "`<smoothly_flowing_prose>` 태그 안에 써"
3. **프롬프트 스타일을 출력 스타일과 맞춰라:** 프롬프트에서 마크다운을 빼면 출력의 마크다운도 준다.
4. 세부 포맷은 명시적 블록으로 지정 (불릿 남발 금지 등).

> 수식이 기본 LaTeX로 나온다. 평문을 원하면 "LaTeX/MathJax 쓰지 말고 일반 텍스트로" 명시.

## 단계적 사고 / Thinking

- **일반 지시 > 세세한 단계 지정.** "충분히 생각해(think thoroughly)"가 직접 짠 단계보다 나을 때가 많다.
- thinking이 꺼진 경우: `<thinking>` … `<answer>` 태그로 추론과 답을 분리해 유도.
- **자기검증 추가:** "끝내기 전에 [기준]에 비춰 답을 검증해" → 코딩·수학 오류를 잘 잡는다.
- 멀티샷 예시 안에 `<thinking>`을 넣으면 추론 패턴을 일반화한다.

## 도구 사용 (Tool use)

- 행동을 원하면 명시적으로. "개선안 제안해줄래?" → 제안만 함 / "이 함수 성능 개선하게 수정해" → 실제 수정.
- 독립적인 도구 호출은 **병렬로** 시키면 빨라진다.

## 복잡한 프롬프트는 체이닝 (Chaining)

가장 흔한 패턴 = **자기교정:** 초안 생성 → 기준으로 리뷰 → 리뷰 기반 수정.
각 단계를 별도 호출로 두면 중간 출력을 검사·로깅·분기할 수 있다.
(GAN 원칙 — 만드는 역할과 채점하는 역할을 분리하라.)
